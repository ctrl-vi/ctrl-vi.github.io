## Overview

![Block diagram of all the modules in the snake game](@assets/snake/top-level.png)
*Fig. 1: Block diagram of all the modules in the snake game.*


Above is the block diagram representing the entire game. The user input informs the pixel/wall drivers for state transitions. The length counter keeps track of the length of the snake. We then have the apple selector, which will tell us which of the pixels is an apple (utilizing a random number from the LSFR). We also have the scoreboard modules, which combine a counter and combinational logic to display a hex display signal for each (relevant) hex module. Then, we have the LED Board driver (which is preexisting) that takes in the green and red LED outputs from the drivers. To enable the ability to select a new apple/increase the length/increase the score, we have a rising edge detector so that only when the length INITIALLY increases do those things occur. There is also a clock divider, as each pixel updates ~256 slower than the fast circuits. 

The playfield module, which comprises pixel and wall drivers, runs at a slower clock speed to ensure the snake doesn't move too fast. Each pixel driver has MANY connections, including to each neighbor to know if they're representing a “head” or “green,” to each other to see if a loss occurred, the length, if there was a length increase from any other modules, the direction, whether another module has reported a loss, and finally the enable signal. We output whether this module is green red, is a head, and if this module indicates a loss or a length increase.

## User Input

This is the thing to filter the inputs of the keys and then utilize them with the following FSM to generate directions (the leftmost keypress takes priority, and the transition is the same coming from all states). We also store the direction, as if the keys are released, we just stay at the same state. The states represent the directions, with 00 being up, 01 being right, and so on (I initially used enums for this, but they couldn’t be used easily with ModelSim).

To determine what the Apple selector is selecting, we need a piece of combinational logic: an 8:256 multiplexer. Combining 17 4:16 multiplexers can achieve this by using one as a selector between the other 16.

## Apple Selector

This module will choose a new VALID square to put the apple, informed on whether it has been enabled. Note the combinational logic to decide whether to draw a new value; it calculates if the selected pixel is already occupied (with a green pixel). If not, it will then utilize that value. Also, if we are not enabled, we just stay at the current value. Note that this uses the LFSR covered in Lab 7, but with different taps and lengths!

![FSM for the apple selector](@assets/snake/apple_selector_fsm.png)
*Fig. 8: FSM for the apple selector, showing the transitions between states based on whether a new apple is needed and if the selected pixel is valid.*

![Simulation of the apple selector](@assets/snake/apple_selector_sim.png)

In the simulation, we are selecting 39, but after we enable and choose a new value, we see that it tests 15, 31, and finally 63 before finally arriving at using it. It is checking it against a 256-long bus of values representing the currently unavailable pixels, noting that 15 and 31 are not allowed there, but 63 is!

## Scoreboard 

These simply implement a counter and then decode that output to be usable with the hex display. When enabled, it switches to the next value on the display. Note that two reset signals are parameterized, so the hex displays are dimmed until required. We also only enable the enable bit when we go from 9 to 0 to indicate how it will go to 10; thus, the next display must increment (as in, these are daisy-chainable). 

![FSM for the scoreboard](@assets/snake/scoreboard_fsm.png)
*Fig. 10: FSM for the scoreboard, showing the transitions between states based on whether to increment the score and if the counter has reached its maximum value.*

![Simulation of the scoreboard](@assets/snake/scoreboard_sim.png)

As we see in the simulation (which connects HEX2 to the carry-out of HEX1), we can increment multiple times, and then the counter will reset to 0, and then the carry-out will cause HEX2 to increment.

## Pixel Module

The pixel module takes in many inputs and then synthesizes them to move around the following FSM. The important states are OFF, APPLE, HEAD, BODY, and LOSS. If we get the signal that this module is an APPLE, then we transition to being an apple. The “priority 2” is just to stop me from having to write `<all the conditions that would cause OFF to go to HEAD> & (APPLE_ID == apple_selector)`. The transition from OFF to HEAD is decided on if the neighbor is a head heading “towards” this pixel, and if this is an apple (APPLE_ID is decided by a parameter fed in) then we also increase in length at that moment. Similarly, if this is an established APPLE and becomes the head, it is also a length increase. For HEAD to LOSS, if this is a head and we head towards an enabled pixel, then this is a loss. Otherwise, we progress through all the states for the body (whose length is dictated by the length passed in), and their progression is paused when the length is increased from any pixel.

## Playfield Module

This simply is a way to compartmentalize the generation of the different pixel modules into a form that's easy to feed the output of to the LED boards, and to test it. It uses generate blocks to connect each pixel to its neighbors and the appropriate inputs. We also make sure to include the tie-ins for each of the switches, to control whether obstacles appear (with 4 different switches required)/

![Simulation of the playfield module](@assets/snake/playfield_sim.png)
*Fig. 16: Simulation of the playfield module, showing the behavior of the module during operation.*

Here is the playfield waveform. Again, way less conducive to being a photo, but when you scroll through the waveform, you can see that the on bits change correctly, and you see a string of 1's in the green pixels' 2d vector that move around as the update changes. The apple also is eaten when the head hits it. Finally, we show the ability to fully enable and fully disable the obstacles, and the end-game screen with all 1's in green.

## Top-level Module

Our final step is to combine everything into one module. This simply involves ensuring we have the right logic buses/wires and then routing them correctly (which took a while!)

![Simulation of the top-level module](@assets/snake/top_level_sim.png)
*Fig. 17: Simulation of the top-level module, showing the behavior of the module during operation.*

We finally have an example of the game being played. We use the reset switch to start the game, use the keys to change direction, and finally collect apples (since the LFSR is deterministic, we just need to head to where we know it will be). Finally, we enable all the walls and then play again. Everything works, including the blackout of the screen during a loss.

## Testing

The testing was first done in ModelSim, using the waveforms and seeing that they conform to expectations. This was done module by module, then in totality in the top-level design. Afterward, I would then actually play the game, doing all the edge cases you'd not normally expect in normal play (pressing multiple keys, getting points, coincidentally getting an apple directly in front of me, enabling walls DURING play, resetting in the middle of play, rapidly switching reset and enable walls). I also thought about how “random” the movement of the APPLE was, to see if the LFSR is truly scrolling through all of its valid values. After lots of small logic error rectification, we had no strange/unexpected behavior.
